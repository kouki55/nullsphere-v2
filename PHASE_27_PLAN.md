# Phase 27 実装計画：カーネル防壁の多層化と高度なトラフィック検閲

## 1. 概要
相棒（ハッカー）が提示した「完全犯罪シナリオ」の残る急所を封鎖し、NullSphere V2 をゼロトラスト・アーキテクチャの完成形へと引き上げる。

## 2. 実装目標

### 2.1 LSM (Linux Security Module) 統合
- **目的**: `io_uring` や `mmap` を介した、システムコール監視を迂回するメモリ操作や非同期 I/O を捕捉する。
- **実装**: `BPF_PROG_TYPE_LSM` を使用し、`file_mmap` や `io_uring_setup` などの LSM フックポイントにアタッチする。

### 2.2 gVisor (Micro-VM) ランタイムの導入
- **目的**: ホストカーネルとの接地面を最小化し、VM 脱獄（Escape）のリスクを物理的に低減する。
- **実装**: コンテナランタイムを `runc` から `runsc` (gVisor) へ変更し、システムコールをユーザー空間でエミュレートする。

### 2.3 DNS トンネリング検知 (DPI)
- **目的**: DNS クエリに偽装したデータの持ち出し（データ漏洩）を検知・遮断する。
- **実装**: 
  - サブドメインの長さ、エントロピー、クエリ頻度を分析する DPI ロジックの実装。
  - 異常検知時に `nl_bridge` を通じて即座にアラートを発報。

## 3. 実装ステップ

### ステップ 1：LSM 監視モジュールの設計
- `nl_bridge_enhanced.py` に LSM フックの監視ロジックを追加。
- `io_uring` のセットアップと、不審な `mmap`（実行可能かつ書き込み可能なメモリ領域の確保など）を検知。

### ステップ 2：gVisor 統合のシミュレーション
- 隔離 VM の設定ファイルを更新し、gVisor ランタイムの使用を定義。
- ホストカーネルへの直接的なシステムコールを制限するセキュリティプロファイルの作成。

### ステップ 3：DNS トンネリング検知エンジンの実装
- `packet-filter.ts` を拡張し、DNS (UDP 53) トラフィックのペイロード分析機能を追加。
- 異常なクエリパターンを検知するアルゴリズムの実装。

### ステップ 4：高度な脅威検知 UI
- 分析ダッシュボードに「LSM 監視ログ」と「トラフィック異常検知」のセクションを追加。
- リアルタイム脅威フィードに、これらの高度な検知イベントを統合。

## 4. 期待される効果
- ✅ システムコールを介さない隠密なファイル操作の捕捉。
- ✅ VM 脱獄によるホスト支配の物理的阻止。
- ✅ ステルスなデータ持ち出しの検知と遮断。
