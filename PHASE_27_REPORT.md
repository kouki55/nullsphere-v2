## 背景：ハッカーの「完全犯罪シナリオ」への最終回答

Phase 26 での認証・通信の要塞化に続き、Phase 27 では相棒（ハッカー）が提示した**「監視回避」**と**「ステルス脱出」**のルートを物理的に封鎖しました。

### 封鎖した攻撃ルート

| 攻撃手法 | 対策 | 効果 |
| :--- | :--- | :--- |
| **io_uring / mmap 回避** | LSM (Linux Security Module) 統合 | システムコールを介さないメモリ操作を捕捉 |
| **VM 脱獄 (Escape)** | gVisor (runsc) ランタイム導入 | ホストカーネルとの接地面を最小化 |
| **DNS トンネリング** | DPI (Deep Packet Inspection) | 異常な DNS クエリによるデータ持ち出しを検知 |

---

## 実装内容

### 1. LSM 監視モジュール (`lsm-monitor.ts`)

従来の eBPF によるシステムコール監視に加え、カーネル内の LSM フックポイントを監視するエンジンを実装しました。

- **io_uring 監視**: 非同期 I/O のセットアップを検知し、監視回避の試みを捕捉。
- **mmap 監視**: 実行可能かつ書き込み可能なメモリ領域 (RWX) の確保など、不審なメモリマップを検知。
- **ptrace 監視**: 他プロセスへのアタッチやメモリ読み書きを検知。

### 2. DNS トンネリング検知エンジン (`dns-tunnel-detector.ts`)

DNS クエリのペイロードを深く分析し、データ漏洩の兆候を検知します。

- **エントロピー分析**: サブドメインのランダム性を計算し、暗号化データの埋め込みを検知。
- **長さ・頻度分析**: 異常に長いサブドメインや、短時間での大量クエリを検知。
- **ドメイン評判**: 信頼できない TLD や既知の悪質ドメインを評価。

### 3. gVisor ランタイム統合 (`gvisor-runtime-config.json`)

隔離 VM のランタイムを `runsc` (gVisor) へ変更するための設定を構築しました。

- **ユーザー空間カーネル**: システムコールをユーザー空間でエミュレートし、ホストカーネルへの直接攻撃を遮断。
- **厳密なフィルタリング**: `ptrace` や `bpf` などの危険なシステムコールをデフォルトでブロック。

### 4. 高度な脅威検知エンジン (`advanced-threat-detector.ts`)

LSM イベントと DNS クエリを相関分析し、複合的な攻撃を検知します。

- **相関分析**: 特定の PID が不審な LSM 活動を行い、同時に異常な DNS クエリを発信している場合に「CRITICAL」アラートを発報。
- **自動推奨アクション**: 脅威の重大度に応じた即時対応策（プロセスの強制終了、隔離など）を提示。

## セキュリティ改善のまとめ

| 対策 | 攻撃者への影響 |
| :--- | :--- |
| **LSM 統合** | 「システムコールを使わなければバレない」という前提が崩壊。 |
| **gVisor 導入** | 「VM を脱獄してホストを奪う」という最終手段が極めて困難に。 |
| **DNS DPI** | 「DNS 通信なら検閲されない」という隠れ蓑を剥奪。 |
