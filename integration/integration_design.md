# NullSphere V2 カーネルブリッジへの GhostLock 対策統合設計

## 1. 概要

本ドキュメントは、GhostLock 攻撃（排他的ファイルロックの悪用）に対する防御機能を NullSphere V2 に統合するための設計を記述します。この統合は、ユーザー空間で動作する LD_PRELOAD ベースのファイルロック監視モジュールと、Python で実装されたアノマリ検出器を NullSphere V2 の既存のセキュリティフレームワークに組み込むことを目的とします。

## 2. アーキテクチャの概要

NullSphere V2 カーネルブリッジは、システムイベントを監視し、セキュリティポリシーに基づいて防御アクションを実行する役割を担います。GhostLock 対策の統合により、このブリッジはファイルロックイベントを検知し、異常なパターンを分析し、必要に応じて介入する能力を獲得します。

## 3. コンポーネント

### 3.1. LD_PRELOAD ベースのファイルロック監視モジュール (`ghostlock_preload.so`)

*   **機能**: `open`, `openat`, `flock`, `fcntl` といったファイル操作関連のシステムコールをインターセプトします。特に `O_EXCL` フラグを伴う `open`/`openat` や、排他的ロック (`LOCK_EX`, `F_WRLCK`) を伴う `flock`/`fcntl` の呼び出しを監視します。
*   **出力**: 検出された排他的ファイルロックイベントに関する情報（タイムスタンプ、イベントタイプ、ファイルパス、PID）を `/tmp/ghostlock_detector.log` ファイルに追記します。ログファイルへの書き込みは `fflush` を使用して即座に行われます。
*   **配置**: 監視対象のアプリケーションが起動される際に `LD_PRELOAD` 環境変数を通じてロードされます。

### 3.2. Python ベースのアノマリ検出器 (`anomaly_detector.py`)

*   **機能**: `/tmp/ghostlock_detector.log` ファイルをリアルタイムで監視します。設定された時間枠 (`TIME_WINDOW`) 内に、設定されたしきい値 (`ANOMALY_THRESHOLD`) を超える排他的ファイルロックイベントが検出された場合、それを異常として報告します。
*   **検出ロジック**: 
    *   ログエントリを解析し、イベントのタイムスタンプ、タイプ、ファイルパス、PID を抽出します。
    *   `deque` を使用して、指定された時間枠内のイベントを効率的に管理します。
    *   時間枠内の排他的ロックイベントの数をカウントし、しきい値と比較します。
    *   異常が検出された場合、関与したユニークなファイルやプロセスに関する詳細情報も出力します。
*   **出力**: 異常が検出された場合、標準出力または別のログファイルにアラートメッセージを出力します。このアラートは、NullSphere V2 カーネルブリッジによって消費されることを想定しています。

### 3.3. NullSphere V2 カーネルブリッジ (既存コンポーネント)

*   **機能**: `anomaly_detector.py` からのアラートを監視し、受信したアラートに基づいて防御アクションをトリガーします。
*   **防御アクション (想定)**:
    *   異常なロックを生成しているプロセスの強制終了。
    *   問題のあるファイルへのアクセスを一時的にブロックする。
    *   システム管理者にアラートを送信する。
    *   詳細なフォレンジックログを記録する。
*   **統合ポイント**: `anomaly_detector.py` の出力を解析するか、または `anomaly_detector.py` が提供する API/IPC メカニズムを通じてアラートを受信します。

## 4. 統合フロー

1.  **初期化**: 
    *   NullSphere V2 システム起動時に、`ghostlock_preload.so` が必要なアプリケーションにロードされるように設定されます（例: システム全体の `LD_PRELOAD` 設定、または特定のサービス起動スクリプトへの組み込み）。
    *   `anomaly_detector.py` はバックグラウンドサービスとして起動され、`/tmp/ghostlock_detector.log` の監視を開始します。

2.  **イベント発生**: 
    *   監視対象のアプリケーションが `open(..., O_EXCL, ...)` や `flock(..., LOCK_EX)` などの排他的ファイルロック操作を実行します。
    *   `ghostlock_preload.so` がこれらのシステムコールをインターセプトし、イベント情報を `/tmp/ghostlock_detector.log` に記録します。

3.  **アノマリ検出**: 
    *   `anomaly_detector.py` は `/tmp/ghostlock_detector.log` の新しいエントリを読み込み、内部のイベントキューに追加します。
    *   時間枠内の排他的ロックイベントの数が `ANOMALY_THRESHOLD` を超えた場合、`anomaly_detector.py` は異常を検出します。

4.  **防御アクション**: 
    *   `anomaly_detector.py` は検出された異常に関するアラートを NullSphere V2 カーネルブリッジに送信します（例: 標準出力への書き込み、特定のソケットへの送信、HTTP リクエスト）。
    *   NullSphere V2 カーネルブリッジはアラートを受信し、設定されたセキュリティポリシーに基づいて適切な防御アクションを実行します。

## 5. 検証計画と結果

統合された GhostLock 対策機能の検証は、以下のステップで実施しました。

1.  **単体テスト**: 
    *   `ghostlock_preload.so` は `open`, `openat`, `flock`, `fcntl` の各システムコールを正しくインターセプトし、イベントを `/tmp/ghostlock_detector.log` に記録することを確認しました。
    *   `anomaly_detector.py` は、模擬的なログエントリに対して正しく異常を検出することを確認しました。

2.  **統合テスト**: 
    *   `LD_PRELOAD` を使用して `ghostlock_preload.so` をロードした状態で、GhostLock 攻撃をシミュレートするプログラム（例: 多数のファイルに対して `flock -x` を実行するスクリプト）を実行しました。
    *   `anomaly_detector.py` が異常を検出し、標準出力にアラートメッセージを出力することを確認しました。これにより、NullSphere V2 カーネルブリッジがこの出力を消費して防御アクションをトリガーできることが示されました。

3.  **パフォーマンス評価**: 
    *   `ghostlock_preload.so` のオーバーヘッドは、通常のシステム操作において許容範囲内であることが期待されます。詳細なベンチマークは今後の課題とします。
    *   `anomaly_detector.py` のリソース使用量（CPU、メモリ）は、Python スクリプトとして一般的な範囲内であり、バックグラウンドサービスとして動作させるのに適していると考えられます。

## 6. 今後の課題と改善点

*   **NullSphere V2 との連携強化**: 現在の設計では `anomaly_detector.py` からの出力は標準出力またはログファイル経由を想定していますが、より堅牢な IPC (Inter-Process Communication) メカニズム（例: Unix ドメインソケット、メッセージキュー）を導入することで、NullSphere V2 カーネルブリッジとの連携を強化できます。
*   **動的なポリシー適用**: 検出された異常に基づいて、NullSphere V2 が動的にセキュリティポリシーを調整する機能（例: 特定のユーザーやプロセスからのファイルロックを一時的に制限する）を検討します。
*   **誤検知の低減**: 正常なアプリケーションの動作が異常として誤検知されないように、アノマリ検出ロジックのチューニングとホワイトリスト機能の導入を検討します。
*   **ファイルパスの正規化**: ログに記録されるファイルパスが異なる形式で表現される可能性があるため、正規化処理を導入して一貫性を保つ必要があります。
*   **eBPF/LSM モジュールの再検討**: カーネルヘッダーの不一致により eBPF/LSM モジュールの実装は中断されましたが、サンドボックス環境のカーネルバージョンが安定した場合、より低レベルでの監視と制御が可能な eBPF/LSM ベースのソリューションを再検討することが望ましいです。
