# Phase 26 実装計画：ゼロトラスト・アーキテクチャへの転換

## 1. 概要
ハッカーによる「完全犯罪シナリオ」を封鎖するため、システムの認証・認可・通信のすべてを「ゼロトラスト（何も信頼しない）」モデルへと転換する。

## 2. 攻撃ルートと対策

### 🕳️ ルート1：API 認証バイパス
- **現状**: 開発の利便性のために `publicProcedure` が多用され、認証なしでカーネル操作が可能。
- **対策**: 
  - すべての機密操作（カーネル設定変更、ログ取得等）を `protectedProcedure` または `adminProcedure` に戻す。
  - JWT トークンの検証を厳格化し、有効期限と権限（RBAC）をチェックする。

### 🕳️ ルート2：内部通信ポート（TCP 9998）の飽和と偽装
- **現状**: `nl_bridge` と `kernel-bridge` 間の通信が平文かつ無認証。
- **対策**:
  - **mTLS (相互TLS)** の導入。サーバーとクライアント（Python ブリッジ）の両方が証明書を提示し、認証された場合のみ通信を許可。
  - **レートリミット**: 単位時間あたりのパケット数を制限し、飽和攻撃（DoS）を検知・遮断。

## 3. 実装ステップ

### ステップ 1: 認証の再要塞化
- `server/_core/trpc.ts` の見直し。
- `server/routers/` 内の全ルーターをスキャンし、`publicProcedure` を適切な保護レベルに引き上げ。

### ステップ 2: mTLS 基盤の構築
- 自己署名認証局 (CA) の作成。
- サーバー用証明書とクライアント（ブリッジ）用証明書の生成。
- `server/kernel-bridge.ts` を TLS サーバーにアップグレード。
- `nl_bridge_enhanced.py` を TLS クライアントにアップグレード。

### ステップ 3: レートリミットの実装
- `kernel-bridge.ts` にトークンバケットアルゴリズムによる流量制御を追加。

## 4. 期待される効果
- 未認証ユーザーによるシステム操作の完全排除。
- 内部ネットワーク内でのなりすまし攻撃の無効化。
- ログ飽和による監視回避の困難化。
