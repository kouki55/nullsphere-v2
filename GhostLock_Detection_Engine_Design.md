# GhostLock 専用検知エンジン 設計書

## 1. 背景と課題

GhostLock攻撃は、Windowsの `CreateFileW` APIにおいて `dwShareMode` を `0x00000000`（排他的アクセス）に設定し、大量のファイルをロックすることで業務を妨害する攻撃手法です。

**既存のセキュリティソフト（EDR/AV）が検知できない理由：**
1.  **ディスクへの書き込みがない**: ファイルの内容を変更・暗号化しないため、ランサムウェア特有の「大量のファイル書き換え」や「エントロピーの変化」を検知するロジックが機能しません。
2.  **正規のAPI呼び出し**: `CreateFileW` による排他ロック自体は、データベースやログファイルなど、正規のアプリケーションでも一般的に使用される正当な操作です。
3.  **シグネチャが存在しない**: 特定のマルウェアバイナリに依存せず、PowerShellスクリプトや正規ツールの悪用（LoL: Living off the Land）でも実行可能なため、静的解析をすり抜けます。

## 2. 検知アプローチ（振る舞いベース）

既存のセキュリティソフトの盲点を突くため、NullSphere V2では「ファイル操作の振る舞いパターン」に特化した専用のヒューリスティック検知エンジンを実装します。

以下の3つの指標（Indicators of Attack: IoA）を組み合わせてスコアリングし、GhostLock攻撃を特定します。

### 2.1. 指標1: 排他的ロックの頻度異常 (High Frequency Exclusive Locking)
*   **ロジック**: 単一のプロセスが、極めて短い時間枠（例: 10秒間）内に、多数の異なるファイルに対して排他的ロック（ShareMode = 0）を要求したか。
*   **しきい値**: 10秒間に50ファイル以上（環境に応じて調整可能）。
*   **理由**: 正規のアプリケーションが短時間に数十〜数百のファイルを排他的に開くことは稀です。

### 2.2. 指標2: ロック保持時間の異常 (Prolonged Lock without I/O)
*   **ロジック**: ファイルを排他的に開いた後、読み取り（Read）や書き込み（Write）のI/O操作を一切行わずに、長時間（例: 5秒以上）ロックを保持し続けているか。
*   **理由**: 正規のアプリケーションは、ファイルを開いた直後に読み書きを行い、速やかに閉じるのが一般的です。I/Oを伴わない長時間のロック保持は、単なる「アクセス妨害」を目的としている可能性が高いです。

### 2.3. 指標3: ディレクトリ走査パターン (Recursive Directory Traversal)
*   **ロジック**: 同一プロセスによる排他ロックの対象ファイルが、特定のディレクトリツリーを再帰的に舐めるようなパターン（例: `C:\Data\A.txt` -> `C:\Data\B.txt` -> `C:\Data\Sub\C.txt`）になっているか。
*   **理由**: GhostLock攻撃は通常、スクリプト等を用いて共有フォルダ全体を再帰的にロックします。

## 3. 実装設計

### 3.1. Windows版 (C# / ETW)
*   **イベントソース**: ETW (Event Tracing for Windows) の `Microsoft-Windows-Kernel-File` プロバイダー。
*   **監視対象イベント**:
    *   `FileCreate` (EventID 4656/4663相当): `ShareMode` が `0` であることを確認。
    *   `FileRead` / `FileWrite`: ロック後のI/Oアクティビティを確認（指標2用）。
    *   `FileClose`: ロック解放を確認。
*   **状態管理**: プロセスIDごとに、現在ロック中のファイルリスト、ロック取得時刻、I/O発生フラグをメモリ上（ConcurrentDictionary等）で管理します。

### 3.2. Linux版 (Python / LD_PRELOAD または eBPF)
*   **イベントソース**: `LD_PRELOAD` による `open`, `openat`, `flock`, `fcntl` のフック、または eBPF/LSM。
*   **監視対象イベント**:
    *   `open` / `openat`: `O_EXCL` フラグや、その後の `flock(LOCK_EX)` 呼び出し。
    *   `read` / `write`: I/Oアクティビティの確認。
*   **状態管理**: Windows版と同様に、プロセスごとのロック状態とI/O履歴を管理します。

## 4. アラートと対応

検知エンジンが上記の指標に基づいてスコアを計算し、しきい値を超えた場合、以下の対応を自動的に行います。

1.  **アラート発報**: NullSphere V2のバックエンドに重大度「High」のアラートを送信。
2.  **プロセス強制終了 (オプション)**: 被害の拡大を防ぐため、該当プロセスを `KillProcess` API（Windows）または `kill -9`（Linux）で強制終了します。
3.  **ロック強制解除 (オプション)**: 可能であれば、WindowsのRestart Manager API等を使用して、対象プロセスが保持しているハンドルを強制的に閉じます。
