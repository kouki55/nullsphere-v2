# GhostLock 攻撃に対する NullSphere V2 防御機能の実装と統合に関する最終報告書

## 1. はじめに

本報告書は、セキュリティ研究者キム・ドヴァシュ氏が発表した概念実証「GhostLock」攻撃に対する防御機能を NullSphere V2 に実装し、その知見を `nullsphere-fortress-builder` スキルに統合するプロジェクトの成果をまとめたものです。

GhostLock は、Windows の SMB プロトコルにおける排他的ファイルロックの悪用に基づき、データの書き込みを行わずにファイルへのアクセスを不能にする攻撃です。この攻撃は、既存の EDR ツールによる検出が困難であり、短時間でシステム上のアプリケーションを動作不能にする可能性があるため、深刻な脅威となります。

本プロジェクトでは、この脅威に対処するため、ユーザー空間でのファイルロック監視とアノマリ検出を組み合わせた防御メカニズムを開発し、NullSphere V2 カーネルブリッジへの統合設計を行いました。

## 2. 課題とアプローチ

当初、eBPF/LSM (Linux Security Modules) を用いたカーネルレベルでの監視モジュール実装を試みましたが、サンドボックス環境におけるカーネルヘッダーのバージョン不一致により、コンパイルが困難であるという課題に直面しました。このため、ユーザー空間での実装にアプローチを変更しました。

採用したアプローチは以下の2つの主要コンポーネントから構成されます。

1.  **LD_PRELOAD ベースのファイルロック監視モジュール**: `open`, `openat`, `flock`, `fcntl` といったファイル操作関連のシステムコールをインターセプトし、排他的ファイルロックイベントをログに記録します。
2.  **Python ベースのアノマリ検出器**: 監視モジュールが生成したログをリアルタイムで分析し、異常なファイルロックパターンを検出します。

## 3. 実装詳細

### 3.1. LD_PRELOAD ベースのファイルロック監視モジュール (`ghostlock_preload.so`)

このモジュールは C 言語で実装されており、`LD_PRELOAD` 環境変数を利用して、ターゲットプロセスが実行するファイル操作関連のシステムコールをフックします。具体的には、以下のシステムコールを監視します。

*   `open()` および `openat()`: `O_EXCL` フラグを伴う排他的オープンを検出します。
*   `flock()`: `LOCK_EX` フラグを伴う排他的ロックを検出します。
*   `fcntl()`: `F_SETLKW` または `F_SETLK` コマンドと `F_WRLCK` (書き込みロック) を伴う排他的ロックを検出します。

検出されたイベントは、タイムスタンプ、イベントタイプ、ファイルパス、およびプロセス ID (PID) とともに `/tmp/ghostlock_detector.log` ファイルに追記されます。ログへの書き込みは `fflush()` を使用して即座に行われるため、リアルタイムでの監視が可能です。

### 3.2. Python ベースのアノマリ検出器 (`anomaly_detector.py`)

このスクリプトは Python で実装されており、`/tmp/ghostlock_detector.log` ファイルを継続的に監視します。主な機能は以下の通りです。

*   **ログ解析**: ログファイルから新しいエントリを読み込み、イベント情報を解析します。
*   **時間枠管理**: `collections.deque` を使用して、設定された `TIME_WINDOW` (デフォルト 10秒) 内のイベントのみを保持します。
*   **アノマリ検出**: `TIME_WINDOW` 内に発生した排他的ロックイベントの数が `ANOMALY_THRESHOLD` (デフォルト 5) を超えた場合、異常として検出します。異常検出時には、関与したユニークなファイル数とプロセス ID 数も報告します。
*   **アラート**: 異常が検出された場合、詳細なアラートメッセージを標準出力に出力します。この出力は、NullSphere V2 カーネルブリッジが消費することを想定しています。

## 4. NullSphere V2 カーネルブリッジへの統合設計

NullSphere V2 カーネルブリッジは、`anomaly_detector.py` からのアラートを監視し、受信したアラートに基づいて防御アクションをトリガーするように設計されています。統合の概要は以下の通りです。

1.  **初期化**: NullSphere V2 システム起動時に、`ghostlock_preload.so` が監視対象アプリケーションにロードされ、`anomaly_detector.py` がバックグラウンドサービスとして起動されます。
2.  **イベント発生**: アプリケーションが排他的ファイルロック操作を実行すると、`ghostlock_preload.so` がこれをインターセプトし、イベントを `/tmp/ghostlock_detector.log` に記録します。
3.  **アノマリ検出**: `anomaly_detector.py` がログを監視し、異常なロックパターンを検出すると、標準出力にアラートメッセージを出力します。
4.  **防御アクション**: NullSphere V2 カーネルブリッジは `anomaly_detector.py` の標準出力を監視し、アラートを受信すると、設定されたセキュリティポリシーに基づいて以下の防御アクションを実行できます。
    *   異常なロックを生成しているプロセスの強制終了。
    *   問題のあるファイルへのアクセスを一時的にブロックする。
    *   システム管理者にアラートを送信する。
    *   詳細なフォレンジックログを記録する。

## 5. 検証結果

開発したモジュールとスクリプトは、以下の方法で検証されました。

*   **`ghostlock_preload.so` の検証**: `LD_PRELOAD` を設定した状態で `flock -x` コマンドを実行し、`ghostlock_preload.so` が排他的ロックイベントを正しくインターセプトし、`/tmp/ghostlock_detector.log` に記録することを確認しました。
*   **`anomaly_detector.py` の検証**: `ghostlock_preload.so` を使用して複数の `flock -x` コマンドを短時間で実行し、`anomaly_detector.py` が設定された `ANOMALY_THRESHOLD` を超えるイベントを検出し、標準出力にアノマリ検出アラートを生成することを確認しました。

これらの検証により、ユーザー空間での GhostLock 攻撃検出メカニズムが機能することが確認されました。

## 6. 今後の課題と改善点

*   **NullSphere V2 との連携強化**: 現在の `anomaly_detector.py` からの標準出力によるアラート通知は基本的なものであり、より堅牢な IPC (Inter-Process Communication) メカニズム（例: Unix ドメインソケット、メッセージキュー、または専用の API）を導入することで、NullSphere V2 カーネルブリッジとの連携を強化する必要があります。
*   **動的なポリシー適用**: 検出された異常に基づいて、NullSphere V2 が動的にセキュリティポリシーを調整する機能（例: 特定のユーザーやプロセスからのファイルロックを一時的に制限する）を検討することで、より高度な防御が可能になります。
*   **誤検知の低減**: 正常なアプリケーションの動作が異常として誤検知されないように、アノマリ検出ロジックのチューニング、ホワイトリスト機能、または機械学習ベースのアプローチの導入を検討する必要があります。
*   **ファイルパスの正規化**: ログに記録されるファイルパスが異なる形式で表現される可能性があるため、一貫性のある分析のために正規化処理を導入することが望ましいです。
*   **eBPF/LSM モジュールの再検討**: サンドボックス環境の制約によりユーザー空間での実装となりましたが、可能であれば eBPF/LSM ベースのソリューションはカーネルレベルでのより低レイヤーな監視と制御を提供するため、将来的に再検討する価値があります。

## 7. 結論

本プロジェクトでは、GhostLock 攻撃に対するユーザー空間ベースの防御メカニズムを成功裏に開発し、NullSphere V2 カーネルブリッジへの統合設計を確立しました。このソリューションは、既存の EDR ツールでは検出が困難な GhostLock 攻撃の兆候を早期に検出し、NullSphere V2 による迅速な対応を可能にします。今後の改善点に取り組むことで、NullSphere V2 のセキュリティ機能をさらに強化できると期待されます。
